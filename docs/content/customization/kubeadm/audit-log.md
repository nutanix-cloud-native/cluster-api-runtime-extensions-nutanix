Kubernetes auditing provides a security-relevant, chronological set of records documenting the sequence of actions in a
cluster. The cluster audits the activities generated by users, by applications that use the Kubernetes API, and by the
control plane itself.

The `auditLog` variable configures the Kubernetes API server audit backend and, optionally, the audit policy used by
the cluster. When `auditLog` is not set, API server audit logging is not configured by this customization.

This customization is available when the [provider-specific cluster configuration patch]({{< ref ".." >}}) is included
in the `ClusterClass`.

## Audit log backends

Exactly one of `log` or `webhook` may be set. Setting both is rejected by validation. When neither is set, the
customization is a no-op.

### Log file backend

The `log` backend writes audit events to a file on the control plane host. The API server log file is written to
`/var/log/audit/kube-apiserver-audit.log` by default. Note that every control plane host has its own audit log file, and that file includes only the audit events for that host.

| Field           | Type     | Default             | Description                                                                                             |
| --------------- | -------- | ------------------- | ------------------------------------------------------------------------------------------------------- |
| `mode`          | string   | `blocking`          | One of `batch`, `blocking`, or `blocking-strict`. See [Kubernetes audit backends][k8s-audit-backends].  |
| `path`          | string   | (see above)         | Path to the audit log file on the control plane host.                                                   |
| `maxAge`        | integer  | (kube-apiserver)    | Maximum number of days to retain old audit log files.                                                   |
| `maxBackup`     | integer  | (kube-apiserver)    | Maximum number of audit log files to retain.                                                            |
| `maxSize`       | integer  | (kube-apiserver)    | Maximum size in megabytes of an audit log file before it is rotated.                                    |
| `compress`      | boolean  | `false`             | If `true`, rotated log files are compressed using gzip.                                                 |
| `eventBatching` | object   | -                   | Batching options. May only be set when `mode` is `batch`. See [Event batching](#event-batching) below. |

### Webhook backend

The `webhook` backend sends audit events to an external HTTP endpoint defined by a kubeconfig stored in a Secret.

| Field             | Type     | Default    | Description                                                                                                |
| ----------------- | -------- | ---------- | ---------------------------------------------------------------------------------------------------------- |
| `mode`            | string   | `batch`    | One of `batch`, `blocking`, or `blocking-strict`. See [Kubernetes audit backends][k8s-audit-backends].     |
| `secret.name`     | string   | (required) | Name of a Secret in the Cluster's namespace. The Secret must contain a `kubeconfig` key.                   |
| `initialBackoff`  | duration | -          | Initial backoff duration after the first failed request, for example `10s`.                                |
| `eventBatching`   | object   | -          | Batching options. May only be set when `mode` is `batch`. See [Event batching](#event-batching) below.    |

The Secret referenced by `secret.name` must contain the audit webhook kubeconfig under the `kubeconfig` data key:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: <CLUSTER_NAME>-audit-webhook-kubeconfig
  namespace: <CLUSTER_NAMESPACE>
type: Opaque
stringData:
  kubeconfig: |
    apiVersion: v1
    kind: Config
    clusters:
      - name: audit-webhook
        cluster:
          server: https://audit.example.com/events
    users:
      - name: audit-webhook
        user:
          token: <TOKEN>
    contexts:
      - name: audit-webhook
        context:
          cluster: audit-webhook
          user: audit-webhook
    current-context: audit-webhook
```

### Event batching

When the backend `mode` is `batch`, `eventBatching` tunes how events are buffered and flushed. All fields are optional;
unset fields fall back to the kube-apiserver defaults.

| Field            | Type    | Description                                                                                            |
| ---------------- | ------- | ------------------------------------------------------------------------------------------------------ |
| `bufferSize`     | integer | Number of events to buffer before batching.                                                            |
| `maxSize`        | integer | Maximum number of events in one batch.                                                                 |
| `maxWait`        | integer | Maximum time, in seconds, to wait before unconditionally flushing the batch queue.                     |
| `throttleEnable` | boolean | If `true`, enables throttling of generated batches.                                                    |
| `throttleQPS`    | integer | Maximum average number of batches generated per second.                                                |
| `throttleBurst`  | integer | Maximum number of batches generated at the same moment when the allowed QPS was previously underused. |

See the upstream documentation for the precise semantics of each field: [Kubernetes audit backends][k8s-audit-backends].

## Audit policy

The audit policy is used to filter audit events. For more information, see the [upstream documentation][k8s-audit-policy-docs].

By default, an [audit policy][default-policy] embedded in the controller is used. It is based on the policy recommended by the Kubernetes project, and aims to capture security-relevant events while keeping log volume manageable.

To use a custom audit policy, set `policy.configMap.name` to the name of a ConfigMap in the Cluster's namespace. See the [example](#use-a-custom-audit-policy-from-a-configmap) below.

The
ConfigMap must contain the policy under the `policy.yaml` data key, in YAML or JSON format. `binaryData` is not
supported.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: <CLUSTER_NAME>-audit-policy
  namespace: <CLUSTER_NAMESPACE>
data:
  policy.yaml: |
    apiVersion: audit.k8s.io/v1
    kind: Policy
    omitStages:
      - RequestReceived
    rules:
      - level: Metadata
        resources:
          - group: ""
            resources: ["secrets", "configmaps"]
      - level: RequestResponse
        verbs: ["create", "update", "patch", "delete"]
      - level: Metadata
```

The policy is rendered into `/etc/kubernetes/audit-policy.yaml` on each control plane host and passed to the API
server with `--audit-policy-file`.

## Examples

### Write audit events to a file on the control plane host

This example uses the default audit policy.

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: <NAME>
spec:
  topology:
    variables:
      - name: clusterConfig
        value:
          auditLog:
            log:
              mode: blocking
              maxAge: 30
              maxBackup: 10
              maxSize: 100
              compress: true
```

### Send audit events to a webhook using batch mode

This example uses the default audit policy.

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: <NAME>
spec:
  topology:
    variables:
      - name: clusterConfig
        value:
          auditLog:
            webhook:
              mode: batch
              secret:
                name: <CLUSTER_NAME>-audit-webhook-kubeconfig
              initialBackoff: 10s
              eventBatching:
                bufferSize: 10000
                maxSize: 400
                maxWait: 30
                throttleEnable: true
                throttleQPS: 10
                throttleBurst: 15
```

### Use a custom audit policy from a ConfigMap

Create the ConfigMap as shown in [Audit policy](#audit-policy) above, then reference it from the Cluster:

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: <NAME>
spec:
  topology:
    variables:
      - name: clusterConfig
        value:
          auditLog:
            log: {}
            policy:
              configMap:
                name: <CLUSTER_NAME>-audit-policy
```

## Applied changes

When `auditLog` is set with either backend, the `KubeadmControlPlaneTemplate` is patched so that each control plane
node:

1. Has the audit policy written to `/etc/kubernetes/audit-policy.yaml`.
2. Passes `--audit-policy-file=/etc/kubernetes/audit-policy.yaml` to the API server.
3. For the `log` backend, passes `--audit-log-path`, `--audit-log-mode`, and any of `--audit-log-maxage`,
   `--audit-log-maxbackup`, `--audit-log-maxsize`, `--audit-log-compress`, and the `--audit-log-batch-*` flags derived
   from `eventBatching`. The audit log directory `/var/log/kubernetes/audit` is mounted into the API server pod at
   `/var/log/audit/`.
4. For the `webhook` backend, passes `--audit-webhook-config-file`, `--audit-webhook-mode`, and any of
   `--audit-webhook-initial-backoff` and the `--audit-webhook-batch-*` flags derived from `eventBatching`. The webhook
   kubeconfig is written to `/etc/kubernetes/audit-webhook-kubeconfig.yaml` from the referenced Secret.
